# Plan Verify

`plan.md`에 적어둔 계획대로 실제 작업이 진행됐는지 확인하는 검증 단계다. `harness-verify`가 "코드가 요구사항(prd/spec)과 맞는가"를 본다면, 이 Skill은 "계획한 절차를 그대로 지켰는가"를 본다.

## 실행 순서

1. `plan.md`에서 검증 대상 Phase 섹션(브랜치명, 작업 순서, 커밋 단위, 완료 기준)을 읽는다.
2. 실제 git 히스토리를 확인한다.
   - `git log --oneline`으로 해당 Phase의 커밋들이 실제로 존재하는지, 계획한 커밋 단위 순서(예: 테스트 커밋 → 구현 커밋)와 맞는지 확인.
   - 커밋 메시지가 `CLAUDE.md`의 형식(`[type] Phase N: 내용`, 한글)을 지켰는지 확인.
   - 계획한 브랜치명(`feature/기능명`)으로 작업했는지 확인.
   - TDD 대상 로직이라면 테스트 커밋이 구현 커밋보다 먼저인지 확인 (Red → Green 순서).
3. 완료 기준(예: 특정 pytest 통과, 콘솔 메뉴 동작)이 실제로 충족됐는지 확인한다.
4. 결과를 아래 형식으로 보고한다.
   - 계획 대비 실제 진행: 일치 / 불일치 항목 목록
   - 커밋 단위·메시지 형식 준수 여부
   - 종합 판단: 이 Phase는 계획대로 완료됐다고 볼 수 있는지

## 예외

- 버그 수정(`[fix]`)이나 마이너한 수정(`[chore]`) 커밋은 `plan.md`에 미리 적혀있지 않을 수 있다. 이런 커밋이 계획에 없다는 이유만으로 불일치로 판단하지 않는다. 계획에서 벗어난 것으로 봐야 하는 대상은 `[feat]`/`[test]` 등 실제 기능 구현이 계획한 작업 순서·범위와 다른 경우로 한정한다.

## 언제 사용하는가

- 각 Phase를 `main`에 merge하기 전, `harness-verify`와 함께 실행한다.
- Phase 8(Harness 단계)에서는 지금까지의 모든 Phase에 대해 한 번씩 실행한다.
